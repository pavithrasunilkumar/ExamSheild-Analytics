## Anomaly Injection Methodology

10 centers (5%, randomly selected, seed-fixed) were deliberately given anomalous
operational signatures across multiple correlated dimensions, so ML models have
genuine ground truth to recover — not just a single obvious flag:

| Signal | Normal centers | Anomalous centers |
|---|---|---|
| Total score (mean) | ~260 | ~399 (inflated, tightly clustered) |
| Response time variance | 8–25 (natural spread) | 0.5–2.5 (suspiciously uniform pacing) |
| Changed-answer→correct rate | 0.10–0.45 | 0.70–0.95 (implausibly high) |
| Complaints per center | ~1.7 avg | ~7.2 avg |
| Incidents per center | Poisson(0.3) | Poisson(3) |
| Infrastructure (CCTV/biometric/perimeter) | Better, but NOT deterministic | Weaker, but NOT deterministic |
| Invigilator staffing | Full | ~60% of normal (understaffed) |
| Social media monitoring signals | None | 1–4 threat signals per center |

**Important:** infrastructure weakness is correlated with anomaly status but
deliberately *not* deterministic — this is intentional, so your Phase 4 fairness
audit has a real test to perform (can the model distinguish "genuinely risky" from
"just under-resourced"?) rather than a trivial one.

The `is_synthetic_anomalous_center` flag in `center_infrastructure.csv` is your
**ground truth label** for validating anomaly detection models. Do not feed this
column into training — use it only for post-hoc validation (precision/recall of
your unsupervised models against known injected anomalies).

## Known Limitations (be upfront about these in interviews)

- **440 candidates (2.2%) went unallocated** — a small number of centers ran out of
  generated seat capacity before their assigned candidates could be placed. This is
  a realistic edge case (capacity mismatch) but should be fixed in v2 by either
  increasing rooms-per-center or capping candidate assignment to center capacity.
- Only one exam cycle/session exists — trend analysis (moving averages across
  cycles) needs v2 to add multiple historical cycles (2023–2026).
- Chain-of-custody tables (`question_paper_packet`, `transport_log`,
  `storage_facility`, `storage_record`) are not yet populated — planned for v2
  alongside `coaching_center` and `device_registry`.
- `audit_trail` (row-level change log) is not populated — this is an operational
  logging table that would be populated by the application layer, not synthesized.

## Reproducing / Regenerating

```bash
pip install faker pandas numpy --break-system-packages
python3 generate_data.py
```
Change `N_CANDIDATES`, `N_CENTERS`, or `ANOMALOUS_CENTER_FRACTION` at the top of
the script to rescale, and change `SEED` for a different random realization.
